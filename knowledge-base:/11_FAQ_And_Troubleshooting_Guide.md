# TraceMind AI - FAQ and Troubleshooting Guide

## 1. Purpose

This document provides answers to common operational, support, and architecture questions encountered while working with the TraceMind AI platform.

The objective is to:

* Reduce investigation time
* Provide consistent answers
* Support onboarding
* Improve self-service troubleshooting
* Serve as a high-value knowledge source for AI-assisted support

This document is intended for:

* Support Engineers
* Developers
* Architects
* Operations Teams

---

# 2. Frequently Asked Questions

---

## Q1. What is TraceMind AI?

### Answer

TraceMind AI is an AI-powered Production Support Assistant that enables users to investigate production issues using natural language.

The platform combines:

* Operational facts from Splunk
* Enterprise knowledge from the Knowledge Base
* MCP-based tool integrations
* Claude Desktop as the conversational interface

Its goal is to reduce Mean Time to Resolution (MTTR) and simplify production support.

---

## Q2. What problem does TraceMind AI solve?

### Answer

Modern distributed systems generate large volumes of logs and events.

Investigating failures often requires:

* Searching logs
* Correlating events
* Understanding service interactions
* Reviewing runbooks

TraceMind AI simplifies this process by providing a conversational interface that retrieves and explains operational data.

---

## Q3. What is the primary user interface?

### Answer

The primary user interface is:

```text
Claude Desktop
```

Users ask questions in natural language and Claude invokes MCP tools to retrieve operational information.

---

## Q4. What is MCP?

### Answer

MCP (Model Context Protocol) is a standardized protocol that allows AI models to interact with external tools and systems.

Within TraceMind AI, MCP enables Claude Desktop to:

* Discover tools
* Invoke tools
* Retrieve operational facts
* Access enterprise knowledge

---

## Q5. What is the MCP Server?

### Answer

The MCP Server is a Spring Boot application that exposes production support capabilities as MCP tools.

It acts as the integration layer between Claude Desktop and backend systems such as Splunk and the Knowledge Base.

---

## Q6. What is Splunk used for?

### Answer

Splunk is the authoritative source of operational facts.

It stores and indexes ProcessLog events generated by business services.

Typical use cases include:

* Job timeline reconstruction
* Failure analysis
* Retry investigations
* DLT investigations

---

## Q7. What questions does Splunk answer?

### Answer

Splunk answers:

```text
WHAT happened?
```

Examples:

* What failed?
* Which service failed?
* When did it fail?
* How many retries occurred?

---

## Q8. What questions does the Knowledge Base answer?

### Answer

The Knowledge Base answers:

```text
WHY did it happen?

WHAT should be done next?
```

Examples:

* Why are retries limited?
* What is the recovery process?
* How should DLT events be handled?

---

## Q9. Why are operational logs not stored in the Knowledge Base?

### Answer

Operational logs:

* Grow continuously
* Change frequently
* Require exact retrieval

Splunk is better suited for operational searches.

The Knowledge Base stores static organizational knowledge such as:

* Runbooks
* SOPs
* Architecture documents
* Policies

---

## Q10. Why is PostgreSQL + pgvector used instead of Qdrant?

### Answer

For the POC:

* PostgreSQL already exists
* Lower operational complexity
* Faster implementation
* Fewer infrastructure components

The design allows future migration if needed.

---

## Q11. What is a Job?

### Answer

A Job represents a single uploaded CSV file.

Example:

```text
JOB-1001
```

A Job acts as the parent entity for all records contained within the file.

---

## Q12. What is a Record?

### Answer

A Record represents a single row within a CSV file.

Example:

```text
REC-2001
```

Each Record is processed independently through the platform.

---

## Q13. What is a Member?

### Answer

A Member represents the business entity being processed.

Example:

```text
MEM-5001
```

Member identifiers are used throughout the platform for traceability.

---

## Q14. Which identifiers should be used during investigations?

### Answer

Preferred identifiers:

```text
jobId

recordId

memberId
```

Database-generated identifiers should not be used for operational investigations.

---

## Q15. How do I investigate a failed job?

### Answer

Recommended sequence:

```text
get_job_summary
      │
get_job_timeline
      │
get_retry_events
      │
get_dlt_events
```

This approach provides complete visibility into job execution.

---

## Q16. How do I investigate a failed record?

### Answer

Use:

```text
get_record_timeline
```

This tool reconstructs the complete lifecycle of a specific record.

---

## Q17. How do I identify retry activity?

### Answer

Use:

```text
get_retry_events
```

Retry events indicate transient technical failures that triggered automatic recovery attempts.

---

## Q18. How do I identify DLT activity?

### Answer

Use:

```text
get_dlt_events
```

DLT events indicate that retry attempts were exhausted and processing could not recover automatically.

---

## Q19. What is a ProcessLog?

### Answer

A ProcessLog is a structured JSON event emitted by services.

It contains operational information such as:

```text
timestamp
service
status
jobId
recordId
errorType
retryCount
```

ProcessLog events form the foundation of Splunk investigations.

---

## Q20. Why is ProcessLog important?

### Answer

ProcessLog enables:

* Traceability
* Timeline reconstruction
* Failure analysis
* Retry analysis
* AI-assisted support

Without ProcessLog, operational investigations would be significantly more difficult.

# 3. Troubleshooting Guide

---

## Problem: Job Never Completed

### Symptoms

* Job remains in progress
* No completion event observed

### Investigation

```text
get_job_summary
      │
get_job_timeline
```

### Possible Causes

* Service failure
* Retry activity
* DLT event
* Infrastructure issue

---

## Problem: Record Not Processed

### Symptoms

* Record status unchanged
* Missing downstream activity

### Investigation

```text
get_record_timeline
```

### Possible Causes

* Event publication failure
* Consumer failure
* Processing exception

---

## Problem: Email Not Sent

### Symptoms

* User reports missing email

### Investigation

```text
get_failed_emails
      │
get_retry_events
      │
get_dlt_events
```

### Common Causes

* SMTP unavailable
* Network timeout
* Dependency outage

---

## Problem: Print Processing Failed

### Symptoms

* Print request never completed

### Investigation

Review record timeline and print-service events.

### Common Causes

* Document generation failure
* Storage issue
* Processing exception

---

## Problem: Archival Did Not Complete

### Symptoms

* Record remains active
* Archival event missing

### Investigation

Review:

```text
archival-request
```

and archival-service activity.

### Common Causes

* Storage failure
* Archival repository unavailable
* Database issue

---

## Problem: High Retry Volume

### Symptoms

* Increased processing time
* Multiple retry events

### Investigation

```text
get_retry_events
```

### Common Causes

* SMTP instability
* Database latency
* Kafka connectivity issues
* Network interruptions

---

## Problem: High DLT Volume

### Symptoms

* Increased failed processing
* Multiple DLT entries

### Investigation

```text
get_dlt_events
```

### Common Causes

* Infrastructure outage
* Dependency failure
* Misconfiguration
* Persistent technical issues

---

## Problem: Slow Processing

### Symptoms

* Long job completion times

### Investigation

```text
get_processing_duration
```

### Common Causes

* Retry activity
* Kafka backlog
* Service bottlenecks
* Infrastructure degradation

---

## Problem: Service Appears Unhealthy

### Symptoms

* Increased failures
* Processing delays

### Investigation

```text
get_events_by_service
```

### Common Causes

* Resource exhaustion
* Dependency outage
* Configuration issues

---

## Problem: Stage-Specific Failures

### Symptoms

* Failures concentrated in one stage

### Investigation

```text
get_events_by_stage
```

### Common Causes

* Service defects
* Infrastructure instability
* Downstream dependency failures
