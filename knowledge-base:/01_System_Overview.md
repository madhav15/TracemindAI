# TraceMind AI - System Overview

## 1. Purpose

TraceMind AI is an AI-powered Production Support Assistant designed to simplify operational troubleshooting, incident investigation, and system analysis across distributed microservices.

The platform enables support engineers, developers, and architects to interact with production systems using natural language rather than manually searching logs, dashboards, and operational tools.

TraceMind AI combines operational facts from Splunk with enterprise knowledge stored in a vector-enabled knowledge repository to provide accurate, contextual, and actionable responses.

The primary objective is to reduce Mean Time to Resolution (MTTR), improve support efficiency, and accelerate root cause analysis.

---

## 2. Scope

TraceMind AI focuses on production support use cases including:

* Job execution tracking
* Record lifecycle tracing
* Failure investigation
* Retry analysis
* Dead Letter Queue (DLT) analysis
* Service-level event investigation
* Processing duration analysis
* Operational troubleshooting
* Knowledge retrieval from runbooks and documentation

The platform is intended for:

* Support Engineers
* Application Developers
* Technical Leads
* Solution Architects
* Operations Teams

TraceMind AI does not replace monitoring systems, observability platforms, or operational databases. Instead, it acts as an intelligent support layer on top of existing enterprise systems.

---

## 3. Business Problem

Modern enterprise applications typically consist of multiple distributed services communicating through asynchronous messaging platforms.

When failures occur, support teams often need to:

* Access multiple systems
* Search large volumes of logs
* Correlate events across services
* Understand retry behavior
* Identify failure points
* Locate relevant runbooks
* Determine corrective actions

This process is time-consuming, error-prone, and heavily dependent on institutional knowledge.

TraceMind AI addresses these challenges by providing a unified conversational interface capable of:

* Retrieving operational facts
* Correlating events across services
* Accessing enterprise knowledge
* Presenting concise explanations
* Guiding support engineers toward resolution

## 4. High-Level Architecture

TraceMind AI follows a Retrieval-Augmented Production Support architecture.

Operational data and knowledge data are intentionally separated.

### Operational Fact Layer

Production events are generated by microservices and written as structured ProcessLog entries.

These logs are indexed by Splunk and serve as the authoritative source of operational truth.

Examples:

* Job execution history
* Record processing history
* Service failures
* Retry attempts
* DLT events
* Processing durations

### Knowledge Layer

Enterprise knowledge is stored separately within PostgreSQL using pgvector.

Knowledge includes:

* Architecture documents
* SOPs
* Runbooks
* Policies
* FAQs
* Service documentation
* Business glossary entries

Knowledge is embedded using OpenAI embeddings and retrieved through semantic similarity search.

### AI Interaction Layer

Users interact with Claude Desktop through MCP (Model Context Protocol).

Claude retrieves:

1. Operational facts from Splunk through MCP tools.
2. Knowledge from PostgreSQL pgvector through knowledge search tools.

Claude then synthesizes the final response.

## 5. Key Components

### Claude Desktop

Primary user interface for natural language interactions.

Responsibilities:

* User conversation management
* MCP tool invocation
* Response generation
* Fact and knowledge synthesis

---

### MCP Server

Spring Boot based MCP server implementing enterprise support tools.

Responsibilities:

* Tool registration
* Tool execution
* Splunk integration
* Knowledge search integration
* Response orchestration

Examples of supported capabilities:

* Job timeline retrieval
* Failure analysis
* Retry investigation
* DLT inspection
* Knowledge retrieval

---

### Splunk

Authoritative operational data source.

Responsibilities:

* Log indexing
* Event search
* Timeline reconstruction
* Failure investigation

Splunk answers:

"What happened?"

Production logs remain exclusively within Splunk and are never embedded into vector storage.

---

### PostgreSQL + pgvector

Enterprise knowledge repository.

Responsibilities:

* Knowledge storage
* Embedding storage
* Semantic search
* Document retrieval

PostgreSQL answers:

"Why did it happen?"

"What should be done next?"

---

### OpenAI Embeddings

Used to generate vector embeddings for knowledge documents.

Embedding Model:

* text-embedding-3-small

Embeddings are generated during knowledge ingestion and stored within PostgreSQL pgvector.

---

### Apache Kafka

Asynchronous communication backbone between services.

Responsibilities:

* Event-driven processing
* Decoupled service communication
* Retry workflows
* DLT handling

## 7. Design Principles

### Operational Truth Remains in Splunk

Production events, logs, retries, and execution history are stored only within Splunk.

No operational logs are embedded into the vector database.

---

### Knowledge and Facts Are Separated

Operational facts and enterprise knowledge serve different purposes and are stored independently.

Splunk provides factual system state.

Knowledge storage provides contextual understanding and recommended actions.

---

### Business Identifiers First

All investigations use business identifiers rather than database-generated identifiers.

Primary identifiers:

* jobId
* recordId
* memberId

This improves traceability and aligns with support workflows.

---

### Minimal Infrastructure Complexity

The architecture intentionally minimizes operational overhead by reusing existing PostgreSQL infrastructure and avoiding unnecessary platform dependencies.

---

### AI as an Augmentation Layer

TraceMind AI does not replace existing enterprise systems.

Instead, it augments them by enabling natural language access to operational data and organizational knowledge.

---

### Explainability

Responses should always be traceable to either:

* Operational facts retrieved from Splunk
* Knowledge retrieved from the enterprise knowledge base

This ensures transparency and trustworthiness.
